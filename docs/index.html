install

Two ways in. uv is the fast one.

Pure Python standard library — nothing to pip-install to run it. Clone and launch, or let uv manage an isolated environment.

uv · recommended
# get uv (Linux & macOS, amd64 + arm64)
curl -LsSf https://astral.sh/uv/install.sh | sh

# install the `voidwalker` command globally
uv tool install .
voidwalker --dry-run   # per-arch install plan
voidwalker selftest    # validate the catalog
checkout · pip
git clone https://github.com/daemon-404/voidwalker.py.git
cd voidwalker.py

# straight from the checkout — no install
python3 voidwalker.py

# …or install it, then use the command
pip install . && voidwalker

Architecture-aware: on ARM, tools with no native binary route to a go/cargo source build instead of grabbing an x86 build that can't run. voidwalker --dry-run prints the exact plan for your machine.

pe workspace

An encrypted memory for the box.

Deploy once with voidwalker env, then pe works on its own — a SQLCipher store for flags, creds, hosts and notes, keyed to the active target. The prompt and tmux bar show which box you're on; pe guide opens the offline handbook.

pe · encrypted store
pe init                       # set a passphrase; create the DB
pe target add Cap --ip 10.10.10.245
pe flag add user 'HTB{...}'
pe cred add -u svc -s 'S3cr3t!'
pe cheat env              # how it all fits together
pe backup                     # snapshot the encrypted DB

Five nouns, the same verbs (add · ls · show · search · rm):

pe target · the box pe flag · user/root pe cred · creds & hashes pe host · hosts & ports pe note · findings

Shell sugar from the env: newbox Cap 10.10.10.245 registers a target and drops you into ~/htb/Cap; scan runs a recorded nmap; Ctrl-X f pastes a flag.

kali docker box

The Parallels VM, containerised.

A headless, repeatable Kali box that reuses VoidWalker as its installer and environment. Baked lean HTB arsenal, native host architecture, the VPN running inside the container. Driven by one wrapper: hbox.

1

Build

host
cd docker && cp .env.example .env
./bin/hbox build   # bakes the arsenal (~20–40m)
2

Enter

host
./bin/hbox up && ./bin/hbox shell
# tmux session; prefix = C-a or C-b
3

Connect

host → vpn inside
# drop .ovpn files in docker/vpn/
./bin/hbox vpn up lab.ovpn
./bin/hbox vpn status
4

Verify

inside the box
./bin/hbox verify   # smoke-test tools
pe init; newbox Cap 10.10.10.245

What persists

Tools are baked into the image by a repeatable build. Your data lives on volumes and survives rebuilds.

Host / volumeIn containerHolds
docker/work//root/htbper-box workdirs
docker/vpn//root/vpn (ro)your .ovpn files
htb-pe-data…/pentest-envencrypted flags/creds DB
htb-state/root/.local/statepe key · zsh history · VPN/runtime state (tmux panes are not persisted)
run it anywhere
# ship the built image (offline, no rebuild)
./bin/hbox save htb-kali.tgz
./bin/hbox load htb-kali.tgz   # on the other machine

# …or push to a registry
./bin/hbox push ghcr.io/you/htb-kali:latest

GUI tools: run Burp Suite & Wireshark natively on macOS. Route container HTTP through Burp with burpenv (sets the proxy to host.docker.internal:8080), or capture with tcpdump -w cap.pcap and open it on the Mac.

macOS note: the HTB VPN runs inside the container because Docker Desktop can't bridge the Mac LAN — that's why the box holds NET_ADMIN and /dev/net/tun.
